# 战场问题清单

> 2026-07-22 审计，只列问题，不包含解决方案。

## P1 — 功能性问题

### 1. `BOW_SAFE_MIN = 0` 导致远程风筝完全失效

**文件：** `js/combat.js:112`

`kiteAsBackline()` 的触发条件是 `dist < BOW_SAFE_MIN`，但值设成了 `0`。`dist`（两点间距）不可能 `< 0`，所以远程兵被近身后不会后撤拉开距离。

- `combat.js:675` 的 `if (dist < BOW_SAFE_MIN)` 分支永不为真
- `combat.js:401-413` 的 `kiteAsBackline()` 函数形同死代码
- 远程兵始终原地站桩输出，即使被突击骑脸

### 2. Lv3 西瓜盾卫失去护盾技能

**文件：** `js/skill_system_v17.js:172` vs `js/fruit_mechanics.js:72`

`fruit_mechanics.js` 在 Lv3 给西瓜盾卫加周期性护盾（`s.level >= 3`），但 `skill_system_v17.js` **完全替换**了 `updateFruitPassiveSkills` 函数。

- v17 版只处理 `hasSkillV17(s)` 即 Lv4+ 的单位（`skillRankV17` 返回 `level < 4 ? 0`）
- v17 **没有调用**旧的 mechanics 函数（直接替换而非包装）
- 加载顺序：`fruit_mechanics → skill_v17`，v17 覆盖后 mechanics 版本彻底消失
- **结果：Lv3 西瓜盾卫（游戏前期最常见的等级）没有任何护盾技能**

### 3. `wallRoleDamageMulV19` 定义了但从未被调用

**文件：** `js/combat_pacing_v19.js:102-110`

定义了按职责返回攻城伤害倍率的函数：

| 职责 | 倍率 |
|------|------|
| 攻城(siege) | ×1.25 |
| 后排(back) | ×0.78 |
| 辅助/控制 | ×0.58 |
| 坦克 | ×0.62 |
| 突击 | ×0.82 |

但 `combat.js` 的 `attackWall()`（第 564 行）硬编码了统一的攻城伤害公式 `(s.level * 1.45 + s.atk * 0.3) * siegeMul * siegeBoost`，完全没有用到这个函数。

- `patchFullSquadWallAttackV19()` 只是设了个 `_fullSquadV19 = true` 标记
- **所有兵种打墙用同一公式**，攻城单位的伤害优势仅靠 `siege` 属性值（如橙子 2.45）而非职责倍率

### 4. 羁绊单位死亡后 Bond 加成永久残留

**文件：** `js/bond_system.js:149-163`

`applyBonds()` 每帧设置 `_bondCritBonus`、`_bondSiegeBoost`、`_bondRangeBonus` 等字段，但**从不清理**。

- 双鲨突击：`_bondCritBonus = 0.20` 在伙伴死亡后永久生效
- 攻城重器：`_bondSiegeBoost = 0.20` 同样残留
- 射手编队：`_bondRangeBonus = 0.10` 不清除
- `iron_wall` 例外（检查 `_bondIronWall` 防重复），但其他 bond 不检查也不清除
- 机制：存活判断只在 `getActiveBonds()` 时做（确认 bond 是否激活），但已设的字段永不会反向清理

## P2 — 逻辑问题

### 5. 近战兵对"身后目标"不回头处理

**文件：** `js/combat.js:667-669`

近战兵在 `attackTarget` 中如果目标在身后（`!isForwardOf`），跳过目标直接 `advanceTowardWall()`：

```js
if (!isForwardOf(s, target)) { advanceTowardWall(s); return; }
moveTowardEnemy(s, target); return;
```

- 敌人穿过兵线后近战兵不再回头追击
- 对突击兵（raider）突破后排尤为关键——一旦越过前排，前排就彻底不管了
- `moveTowardEnemy` 第 388-396 行也有"只前进不后退"逻辑，确认了"单向推进"的意图
- 但如果目标是绕后的远程兵，近战兵也完全不理会

### 6. 点燃伤害立即触发

**文件：** `js/status_engine_v61.js:119`

`applyStatus` 设 `tickTimer = 0`：

```js
se.burning.tickTimer = 0;
```

下一帧 `tickStatus` 检查 `tickTimer <= 0` 为真，立即造成第一次燃烧伤害。

- 2s 点燃实际可造成 3 次伤害（t=0s, 1s, 2s）而非设计的 2 次
- 如果 `tickTimer` 初始化为 `1.0`，则只在 t=1s 和 t=2s 造成伤害
- 当前行为给了点燃一次"瞬间爆发"，偏离了 1tick/s 的设计

### 7. 眩晕结束帧出现"空闲行动"窗口

**文件：** `js/combat.js:1013-1026` 执行顺序

每帧执行顺序：
1. `tickStatus(s, dt)` — 减少眩晕计时器
2. `updateSoldier(s, enemies)` — 检查 `isDisabled(s)` 决定是否跳过行动

如果眩晕计时器恰好在 `tickStatus` 中从 `>0` 减到 `<=0`，同一帧的 `updateSoldier` 看到 `isDisabled = false`，兵会多行动一帧（本应被眩晕）。

- 只有一帧窗口，对实际游戏影响极小
- 但所有状态相关的定时器都有此问题

### 8. 远程和近战攻城等级系数不一致

**文件：** `js/combat.js:492` vs `js/combat.js:564`

- 远程打墙（`rangedAttackWall`）：`s.level * 1.25 + s.atk * 0.3`
- 近战打墙（`attackWall`）：`s.level * 1.45 + s.atk * 0.3`

等级系数差距 16%（1.25 vs 1.45），`atk` 系数相同（0.3）。可能存在设计意图（近战攻墙更快），也可能是不一致残留。

### 9. 自由战场中 wildcard 职责无映射

**文件：** `js/free_battle_v2.js:67`

`freeRole()` 把新职责映射到旧角色名：

```js
if (role === 'shell') return 'tank';
if (role === 'spike') return 'front';
if (role === 'shooter') return 'ranged';
return role; // raider→raider, wildcard→wildcard
```

- wildcard → `'wildcard'`（原样返回）
- `findTargetFree` 中的职责检查：`if (role === 'tank' || role === 'front')`、`if (role === 'raider')`、`if (role === 'ranged')` 都不匹配 `'wildcard'`
- wildcard 单位的索敌/行为在自由战场中走默认路径，没有针对性的行为逻辑

### 10. 攻城排队兵 Y 偏移可能导致 reachedWall 反复进出

**文件：** `js/combat.js:529-533`

排队兵被拉到 `attackY + 16 + row * QROW`。对 player 方，这意味着 Y 更远离敌方城墙方向（更靠近己方）。`reachedWall` 检测可能在边界附近来回切换。

```js
const queueY = s.side === 'player' ? wall.attackY + 16 + row * QROW : wall.attackY - 16 - row * QROW;
s.y += (queueY - s.y) * Math.min(1, dt_global * 7);
```

### 11. 冰镇果汁折扣标记可能悬空

**文件：** `js/economy_balls_v62.js:61`

`_econChillReady` 只在 `cc > state._econLastCounter`（有新召唤操作）时清除。如果 `actionCost` 已经使用了折扣但后续没有新的操作，标记可能残留。

```js
if (cc > state._econLastCounter) {
    const honey = econLevel('sp_refund');
    if (honey > 0 && (state._econRefundCd || 0) <= 0) { grantSp(tier(honey)); state._econRefundCd = 8; }
    state._econChillReady = null; // 只在这里清除
}
```

### 12. `_fruitPatched` 防重复机制仅拦截自身

**文件：** `js/fruit_mechanics.js:156-166`

```js
if (typeof updateCombat !== 'function' || updateCombat._fruitPatched) return;
```

fruit_mechanics 通过 `_fruitPatched` 标记防止自己重复包装，但无法防止其他模块（如 bond_system）的包装链出错。目前执行链正确，但防御机制薄弱。

### 13. 教学关 `enemyBallTimer = 999` 不阻止 `juice_economy` 的敌方自动刷球

**文件：** `js/tutorial_balance.js:70` vs `js/juice_economy.js:249-258`

教学关第 1 关设 `state.enemyBallTimer = 999` 试图禁用敌方刷球，但 `juice_economy.js` 的 `tryEnemyJuiceSummon()` 每 `ENEMY_ACTION_INTERVAL（4s）` 调用 `autoSpawnEnemyPlanBall` 刷球，完全不受 `enemyBallTimer` 控制。

- 教学第 1 关设计的"静谧开局"（只有 2 个敌方兵）可能被 bypass
- 实际是否触发取决于 `state.enemySp` 是否够支付刷球费用（`enemyActionCost()`）
- 教学关设 `state.enemySp = 6`，可能够刷 2-3 轮

## D — 设计文档对照问题

> 对照 `新建 文本文档.txt`（潮战回合制 + 五段战线设计）发现当前实现与设计之间的差距。

### D1. 无回合/潮汐阶段状态机

**设计文档 §2：** 每局由多个 ~20s 潮战回合组成（潮前准备 3s → 潮战 12s → 潮决 2s → 突破 3s → 回卷 2s）。

**现状：** 战斗是连续实时流，没有回合概念。`state.tide.phase` 存在（`'calm'/'surge'`）但只影响移速 ×1.08，不与回合阶段挂钩。`state.round` 状态机不存在。

### D2. 棋盘列 ↔ 战场段 映射断裂

**设计文档 §6：** 棋盘第 1~5 列直接对应五段战线（左浅滩/左潮道/中央深沟/右潮道/右暗流）。列决定出兵到哪一段。

**现状：**
- `free_battle_v2.js:13` — `freeSpawnX()` 用 `stableHash` 伪随机生成 X 坐标，完全忽略棋盘列
- `free_battle_v2.js:314` — 自由战场中所有 soldier 的 `laneIndex` 被设为 `0`
- 只有旧棋盘模式（`main.js`）下 `laneIndex` 随棋盘列生成，但自由战场 bypass 了

### D3. 无每段兵力上限

**设计文档 §4：** 每段最多容纳双方各 2 个战斗单位。全场双方各最多 10 个。

**现状：** `MAX_SOLDIERS = 10` 是全局限制，不分区段。理论上某一段可以有 8 个兵而另一段 0 个。`laneStats` 虽然统计每段兵力但只是展示用，不做硬限制。

### D4. 无中场限制线

**设计文档 §3：** 潮战阶段兵不能越过中线进入敌方半场。突破阶段胜方获得"破潮"状态穿越中线。

**现状：** 全场开放移动。`moveTowardEnemy` 的"只前进不后退"逻辑（`combat.js:391`）只是单向约束，根本不限制前进到哪里。兵可以直接走到敌方城墙下。

### D5. 无突破阶段

**设计文档 §2+§3：** 一段清空对方后进入"突破"——存活兵获得 0.25s 无敌 + 50% 加速，冲过中线撞墙结算。

**现状：** `reachedWall()` 检测到兵到墙就立即 `attackWall()` 或 `rangedAttackWall()`，没有突破阶段。`killSoldier` 后也不会触发"本方该段已清空"的结算。撞墙是持续的、零散的，不是回合驱动的战术事件。

### D6. 城墙血量公式不一致

**设计文档 §5：** `城墙耐久 = 10 + 2 × 关卡号`（第 20 关 = 50）。

**现状：** `config.js:376` — `enemyWallHp = Math.round(140 + (id - 1) * 26)`（第 20 关 = 634）。数值差距约 12 倍。加上 `combat.js` 的伤害公式（`s.level * 1.45 + s.atk * 0.3`）产生的大数值，当前设计是"持续磨血"，文档设计是"有限次突破"。

### D7. 撞墙用连续伤害而非离散破阵值

**设计文档 §5：** 破阵值按等级离散分段——Lv1-2=1, Lv3-4=2, Lv5-7=3，攻城+1，上限 3。

**现状：** 攻城伤害用连续数值公式，一个 Lv4 橙子炮手（atk=12×1.96=23, siege=2.45）打墙伤害约为 `(4*1.45 + 23*0.3) * 2.45 = 31`。当前设计文档中第 10 关总耐久只有 30，等价于当前一次攻击的数值，无法直接移植。

### D8. 跨段攻击非常有限

**设计文档 §3：** 射手可以跨相邻段攻击；旗鱼骑手每回合可横切一次相邻段。

**现状：** `CROSS_LANE_EMERGENCY_RANGE = 120`（`combat.js:112`）是纯距离判断，不区分角色职责。没有角色驱动的跨段攻击逻辑。`findTarget` 的跨路优先级很低（score 加 58-180 惩罚），基本是同路锁敌。

### D9. `wallDamageByLane` 存在但几乎不用

**设计文档 §5：** 五段礁门各自显示局部破损。某段累计承受总耐久的 25% 后进入"破口"状态。

**现状：** `state.wallDamageByLane` 数组在 `attackWall()`（`combat.js:567`）中更新，但只记录数据。没有破口状态、没有段位局部破损判定、没有"破口段下一回合 +1 伤害"的机制。数据存在于渲染层从未使用。

### D10. 自由战场职责映射使用旧名

**设计文档 §7：** 采用 `shell/spike/shooter/raider/wildcard` 职责体系。

**现状：** `free_battle_v2.js:62-66` 的 `freeRole()` 把新职责映射回旧名（`shell→tank, spike→front, shooter→ranged`）。`findTargetFree` 第 100-103 行的索敌权重写死为旧名（`role === 'tank'`, `role === 'raider'` 等），职责定制行为不可靠。

### D11. 辅助单位没有段位检查

**设计文档 §7 约束 4：** 辅助单位不允许单独成为一段的唯一"胜者"。

**现状：** 没有段位胜负判定系统，自然也没有辅助单位是否独占一段的检查。

### D12. 召唤物撞墙没有限制

**设计文档 §7 约束 2+§5 表格：** 召唤物不能撞墙，破阵值为 0。

**现状：** `ramWall()`（`free_battle_v2.js:230`）和 `attackWall()`（`combat.js:543`）没有检查是否是召唤物。如果召唤物到达城墙，它们会和普通兵一样撞墙。

### D13. 控制技能跨回合保留

**设计文档 §7 约束 1：** 控制技能不跨回合保留。

**现状：** 连续实时战斗没有回合概念，所有状态计时器（减速/冰冻/眩晕/点燃）持续运行直到自然结束。虽然没有"跨回合"问题，但也无法实现"每回合重置状态"的设计。

### D14. 教学第 1 关无法体现列→段策略

**设计文档 §6：** 5 列对应 5 段，摆棋有战术意义。

**现状：** `tutorial_balance.js:25-30` 的教学布置（葡萄在 1,1/1,2，西瓜在 2,1，蜜桃在 2,2，橙子在 0,2，奇异果在 0,3）在当前连续战场中没有列相关的战术含义——所有兵从各自的 `freeSpawnX` 随机位置出发。教学无法教"坦克放左路、射手放中路"的概念。
