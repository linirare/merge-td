# 战斗屏 UI/UX 设计规范

> 日期:2026-07-12 · 目标屏:局内战斗(HTML5 canvas,W=480 H=854)· 风格:烫金 Claymorphism(延续已批准的壳屏语言)
> 依据:真实 7 区布局(config.js/layout_v56)+ 玩法克制(config.js ROLE_COUNTER)+ UI 审计硬伤(docs/ui-ux-audit)+ ui-ux-pro-max 基线(Claymorphism / 44px / 4.5:1 / tabular / reduced-motion)。

## 0. 设计定位
战斗屏是玩家**停留最久、也最"程序员美术"**的屏。它是 canvas 矢量绘制(非 DOM,贴不了图),所以设计=**用矢量把"烫金 Claymorphism"语言画到 HUD 上 + 修好可读性/触控/反馈**,而不是塞插画。**单位保留 emoji 大头**(这是刻意的零图片依赖美术身份,别动)。

## 1. 视觉风格(与壳屏统一)
- **烫金 Claymorphism**:HUD 面板=深棕金框卡片(`#3E2716→#241206` 底 + `#F5C242` 2px 金边 + 圆角 12–14 + 双阴影 `0 5px 0 暗金` 立体感)。
- **战场底**:暖亮**分区矢量底**(敌区/战场/我区,见 battle-bg-proto),压暗到不抢前景;非 AI 插画。
- **色板**:金 `#F5C242` 主 · 深棕框 · 敌方玫红 `#C97984` · 我方绿 `#4E9A59` · SP 翡翠 `#2FBF71` · 危险 `#E23B4E`。
- **数字全用 tabular/等宽字形**(SP/HP/计时/伤害),防跳动。

## 2. 七区布局 + 信息层级(真实坐标)
| 区 | Y(canvas) | 设计 |
|---|---|---|
| 顶栏 HUD | 4–36 | 关卡/模式(左金牌)· 计时(中,深青药丸)· 速度/暂停/帮助(右金键)**热区≥44(hitSlop)** |
| 敌方 3×5 | 62–266 | 「敌方·AI」玫红标签,棋盘略压暗(视觉次级) |
| 敌堡城墙 | 278–298 | 金框血条 + **HP 数字压深色 pill(12px)**、按兵线显破损裂纹 |
| 战场 5 线 | 306–538 | 5 兵线明确分隔 + 中线交锋虚线;**克制反馈**是这里的主角(见 §3) |
| 我堡城墙 | 546–566 | 同敌堡,我方绿 |
| 我方 3×5 | 574–778 | **主交互面**(拖拽/合成);拖拽吸附高亮(金=合成/绿=移动) |
| 操作条 SP + 出球 | 786–832(**最底**) | **核心操作·最亮**:左 SP 大等宽字「8/18」+ 下次消耗;右**出球金按钮**(热区扩、`-消耗`红标)。**贴屏幕最底=拇指最顺手** |

**⚠️ 相对现状的布局调整**:现在 SP+出球 在棋盘**上方**(operationY=574),本设计改到棋盘**下方、屏幕最底**。需改 `layout_v56.js` 的 `operationY`/`playerBoardY`/`bottomY` 坐标 + 同步 `input.js`/`juice_economy.js` 的 hit-test 与 `hud_skin.js` 绘制 Y。**动布局、需 combat-baseline 保绿 + 真机核对不重叠。**

**层级原则**:屏幕**最底的 SP+出球最亮最实**(核心操作、拇指区)→ 其上我方棋盘(主交互)→ 战场次之 → 敌方区压暗。玩家视线自然落在"我能操作的地方"。

## 2.5 画布铺满屏幕(对齐导航页)+ 响应式布局 【新增·用户要求】
**问题**:战斗是 canvas,`main.js` 现用 `scale = min(innerW/W, innerH/H) * 0.96`(W=480 H=854 固定)→ 手机上只有 374px 宽 + 上下留 ~150px 黑边,**比 DOM 导航页(满屏)窄一圈、四周留白、不齐**。

**目标**:战斗 canvas **边到边铺满视口**(像导航页那样满宽、尽量无 letterbox),并把多出来的竖向空间**重排给战场**(现在战场仅占屏高 ~21%,太挤)。**"对齐" 和 "布局优化" 是同一件事**——铺满屏时顺势重排 7 区。

**推荐做法(响应式逻辑高度)**:
- 逻辑宽 `W=480` **不变**(横向棋盘/兵线数学不动,风险低)。
- 逻辑高 **改成随设备比例**:`H = Math.round(480 * innerHeight / innerWidth)` → canvas 逻辑比例 = 设备比例 → **无上下黑边、铺满**。
- `scale = innerWidth / W`(满宽,**去掉 0.96**)。
- H 变动 ⇒ **7 区 Y 坐标必须在 resize 时按 H 动态算**(不再是 layout_v56.js 的固定常量):顶栏/敌方棋盘/敌墙=顶部固定高;**战场 = flex,吃掉多出的竖向空间**(变大);我墙/我方棋盘/操作条=底部固定高。所有 draw 读 `LAYOUT.*` 会自动适配。
- **核心重构点 = 把 `LAYOUT` 从"固定常量"改成"resize 时按 W/H 算的函数"**。

**低风险退路(若全响应式太险)**:只去掉 `0.96`(满宽)+ 接受上下黑边,但把黑边染成与壳屏一致的暖深色并居中,让边不突兀。差一档(仍有边),但改动最小。

**涉及文件**:`main.js`(resize/scale→响应式)、`config.js`/`layout_v56.js`(W/H + LAYOUT→动态)、各 draw 函数(只要读 LAYOUT 即自动跟随)。

**验证**:combat-baseline 绿;真机 390 宽 + 桌面 430 宽两档都截图,战斗**铺满、与导航页同宽、无黑边**;战场明显变大;7 区不重叠。

> 承接 §2 的操作条挪底(已实现)。本节与 §2/§3 属同一次战斗屏重排,**建议由同一执行者一并做**,避免多方改 canvas 撞车。

## 3. 把"克制"这套核心玩法可视化(最大设计增量)
玩法精髓是 7 职责石头剪刀布(rush 克 back ×1.5 等),但现在**战斗里几乎看不出谁克谁**。设计:
- 单位交战瞬间,头顶飘 **克制/优势/受制** 标签(用 `roleCounterText`),**按职责配色**(突击黄/后排紫/坦克绿/攻城橙…)。
- 出兵时若"正好克制当前敌方主力",出球按钮闪金光 + 提示,教玩家配克制。
- 敌堡血条按 5 兵线分段显破损,告诉玩家"哪条线快破了"。
→ 让玩家**看得懂为什么赢/输**,而不是瞎堆兵。

## 4. HUD 组件规范(修审计硬伤)
| 组件 | 现状(审计) | 设计 |
|---|---|---|
| 速度/暂停/帮助 | 20–22px 难点 | 视觉可小,**命中热区 hitSlop 扩到 ~44**(已在分支做 B1) |
| 出球按钮 | 23px 高 | 命中区扩(已做);视觉建议 operationH 提到 46+、金厚边 |
| SP 显示 | 尚可 | 大等宽数字 + 「/18」上限 + 下次消耗预览,强调核心资源 |
| 墙血数字 | 浅压浅 1.8:1 | 深色 pill 底 + 浅字 12px(已做) |
| 小字满屏 6–8px | 难读 | 关键信息 ≥11px effective;次要信息折叠 |

## 5. 反馈 / Juice(有基础,补齐)
- **点按反馈**:出球/速度键按下即时形变(现无按下态)+ 可选轻震动(haptic)。
- **战斗前**:菜单→战斗加一个 0.4s「第 N 关 · 准备」过场(现无 loading,慢机黑一下)。
- **Boss 关**:第 5/10/15/20 关加「Boss 登场」半屏亮相 + Boss 单位视觉放大描金。
- **reduced-motion**:系统开减弱动效时,浮动/抖屏/走路停(已在分支做 C)。

## 6. 明确不做
- 不塞 AI 光栅插画进 canvas(风格不符 + 无透明通道,历史教训)。
- 不改战斗逻辑/数值(纯表现层)。
- 单位 emoji 大头保留(美术身份)。

## 7. 落地路径(建议分阶段,每阶段可独立验证 combat-baseline)
1. **HUD 可读性/触控**(顶栏金牌 + SP/出球 + 墙血 pill + tabular 数字)——多数已在分支 ui-fixes-abcd 做了 B/#4。
2. **克制可视化**(§3,飘字+配色)——最大体验增量,纯表现。
3. **分区战场底 + Boss 登场**(§1/§5)——需与另一 agent 的皮方向对齐后做。

## 验证
每阶段:`node test/combat-baseline.js --check` 保绿(纯表现,快照应零变化)+ Playwright 真机 390×844 进战斗截图肉眼核 + 慢网首帧核。
