### 问题1: 后排单位"安全锚"指向己方城墙，导致永不攻城

**文件:** `js/combat.js` 第 300-305 行
**函数:** `advanceTowardWall()`

**问题代码:**
```js
// 后排无前排时：慢速谨慎推进
const slowAdvance = s.side === 'player' ? -1 : 1;
const safeAnchor = s.side === 'player' ? fieldBottom() - 42 : fieldTop() + 42;
s.y += slowAdvance * sspeed * 0.38 * dt_global;          // (A) 前推力
s.y += (clamp(s.y + (safeAnchor - s.y) * 0.12, ...) - s.y) * Math.min(1, dt_global * 4.2);  // (B) 锚拉力
```

**数学推导 (玩家方):**

`fieldBottom()` = 538 (战场底边), `safeAnchor` = 538-42 = **496** (指向己方城墙方向!)

- **(A) 前推力:** -1 × 81 × 0.38 × dt = **-30.8dt** (向上,正确)
- **(B) 锚拉力:** 0.12 × (496 - s.y) × 4.2 × dt = **0.504×(496 - y)×dt**

**净速度计算:**

| 单位Y位置 | 前推力(A) | 锚拉力(B) | 净速度 | 实际方向 |
|-----------|----------|----------|--------|----------|
| 480 (接近敌墙) | -30.8 | +8.1 | **-22.7 px/s** | ↑ 前进 ✅ |
| 435 (平衡点) | -30.8 | +30.8 | **0 px/s** | ⊘ 停滞 ❌ |
| 380 (战场中段) | -30.8 | +58.5 | **+27.7 px/s** | ↓ 后退 ❌ |
| 320 (战场前段) | -30.8 | +88.7 | **+57.9 px/s** | ↓ 快速后退 ❌ |

**结论:** 后排单位在战场中段以 **27.7 px/s 向己方城墙撤退**。平衡点 Y=435，离敌方城墙还差 ~55px 就停住了。后排单位**永远无法靠自己到达敌方城墙**。

**触发条件:** 后排角色(back/support/control/siege) + 同路无前排盟友

**关联症状:** 🎯 "清完敌兵不打城墙"

### 问题3: `updateFruitPassiveSkills` 和 `updateRollingPumpkins` 每帧被调用两次

**文件:** `js/fruit_mechanics.js` 第 155-165行 + `js/combat.js` 第 727-733行

**调用链追踪:**

```
fruit_mechanics.js 将 updateCombat 包装为 fruitUpdateCombat:
  function fruitUpdateCombat() {
      updateFruitPassiveSkills(dt_global);   // ← 第1次!
      updateRollingPumpkins(dt_global);      // ← 第1次!
      oldUpdateCombat();  // 调用 combat.js 原始版本
  }

combat.js 原始 updateCombat 内部:
  if (typeof updateFruitPassiveSkills === 'function')
      updateFruitPassiveSkills(dt_global);   // ← 第2次! (此时函数已定义)
```

虽然 `combat.js` 加载时 `updateFruitPassiveSkills` 还不存在，但 `typeof` 检查在**调用时**求值——此时 `fruit_mechanics.js` 已加载，函数已定义，条件为 true。

**严重度:** 🔴 严重 — 整个被动技能系统的时序被扭曲

### 问题4: applySeparation Y-clamp 将攻城单位弹离城墙 (18px垂直振荡)

**文件:** `js/combat.js` 第 556-560行
**函数:** `applySeparation()`

```js
if (fx || fy) {  // 有任何分离力(含X轴)
    const speed = 42 * dt_global;
    const nextX = a.x + fx * speed;
    a.x = clamp(nextX, a.laneX - FIGHT_X_LEASH, a.laneX + FIGHT_X_LEASH);
    a.y = clamp(a.y + fy * speed * (a.mode === 'siege' || a.mode === 'siege_queue' ? 0 : 1),
                fieldTop(), fieldBottom());  // ← Y被clamp到 ≥296
}
```

**机制:**
- 攻城单位Y = `wall.attackY` = 278 (敌方城墙,在战场上方)
- `fieldTop()` = 296 (战场顶边)
- 当2+攻城单位在同路时,X轴分离力触发 `if(fx||fy)` 条件
- Y被 `clamp(278, 296, 538)` = **296** (弹回战场!)
- 下一帧 `attackWall` 第366行 `s.y = wall.attackY` = 278 (又拉回去)
- **每帧18px垂直振荡**

**触发条件:** 同路≥2个攻城单位 (SIEGE_SLOTS_PER_LANE=3,非常常见)

**关联症状:** 🎯 "动作不顺畅"、"占位乱" — 攻城单位在墙边上下抖动

### 问题5: 后排跟随前排攻城时停在间距外，也永不攻城

**文件:** `js/combat.js` 第 296-298行
**函数:** `advanceTowardWall()`

```js
if (front) {
    const spacing = role === 'siege' ? 78 : role === 'support' ? 70 : 58;
    const desiredY = s.side === 'player' ? front.y + spacing : front.y - spacing;
    s.y += (clamp(desiredY, ...) - s.y) * Math.min(1, dt_global * 4.2);
}
```

**场景:**
1. 前排(坦克)到达敌方城墙,进入 siege 模式,Y=278
2. 后排(射手)通过 `advanceTowardWall` 跟随前排
3. `desiredY` = 278 + 58 = **336** (在城墙后方58px!)
4. 后排停留在Y≈336, `reachedWall()` 检查: `s.y(336) <= wall.attackY(278)`? → **false**
5. **后排永远到不了城墙,只能干站着**

**即使有前排带领,后排也不攻城。** 这个问题独立于问题1——问题1影响"无前排"场景,问题5影响"有前排但前排已在攻城"场景。

**关联症状:** 🎯 "清完敌兵不打城墙"

### 问题11: 近战攻击范围实际为24,非姿态阈值暗示的30 — 6px死区

**文件:** `js/combat.js` 第 444-456行
**函数:** `attackTarget()`

```js
// 第451行: 近战姿态 — 距离>30才追击 (暗示≤30可攻击)
if (melee) { const d = Math.hypot(s.x - target.x, s.y - target.y); if (d > 30) { moveTowardEnemy(s, target); return; } }

// 第455行: 通用射程检查 — fruitRange对近战返回24
if (dist > range) { moveTowardEnemy(s, target); return; }  // range = fruitRange(s) = 24
```

**死区分析 (近战单位, dist=28):**
1. Line 451: d=28 > 30? No → 放行
2. Line 454: isBack? No → 放行
3. Line 455: dist(28) > range(24)? **Yes** → `moveTowardEnemy` (追击!)

**结论:** 近战单位需要追到 dist≤24 才能攻击,而非代码注释暗示的 dist≤30。**6px死区**使近战接战距离缩短20%,单位在24-30px区间反复追击→微量振荡→拖慢战斗节奏。

**根源:** `fruitRange()` 对所有近战角色(tank/front/rush)无条件返回24,覆盖了`ATTACK_RANGES`中的武器特定距离(spear=30, sword=24, shield=22)。后者已成为死代码。

**关联症状:** 🎯 "动作不顺畅" — 近战单位在接战边缘反复微调位置

### 问题12: 被减速的近战单位追不上正常速度目标 — 被无限风筝

**文件:** `js/combat.js` 第 253-259行 + `js/status_engine_v61.js` 第 194-203行

**场景:** 椰子守卫(Lv1, move=72)被冰梨减速(effectiveMul=0.52)

- 目标正常移速(葡萄射手 move=86): `SOLDIER_SPEED * 86/92` = 92 * 0.935 = **86 px/s**

这使减速不仅仅是"变慢",而是让近战单位**彻底丧失接战能力**。与问题2(速度下限丢失)叠加后尤为严重。

**关联症状:** 🎯 "有敌兵不攻击敌兵"、"原地踏步" — 追不上所以永远在追

## 🎨 视觉表现层问题 (第4轮 — 战斗反馈/爽感/差异化)

> **审查范围:** 渲染管线的7层覆盖、动画系统、视觉特效、单位差异化

### 渲染架构概览

项目有7+个渲染覆盖层,按加载顺序堆叠:
```
render.js → skin.js → battle_skin.js → combat_clarity.js 
→ stickman_render_v60.js → fruit_skin.js → hud_skin.js
```
外加两个独立的VFX系统: `juice.js` + `juice_absorb_v16.js`

---

### 🔴 视觉严重问题

#### 视1: 两个果汁VFX系统完全重复,每个命中产生双倍特效

**文件:** `js/juice.js` + `js/juice_absorb_v16.js`

两个文件各自维护独立的特效数组(`state.juice` vs `state.juiceV16`),各自包装 `attackTarget`/`attackWall`/`tryMerge`,各自生成光束/斩击/火花/冲击波/文本。**同一个命中事件触发两套完全重叠的特效。** 这不仅浪费性能,还让特效看起来"脏乱"而非"爽快"。

---

#### 视2: 单位移动无插值 — 逐帧"闪现"而非平滑行走

**文件:** 所有渲染器

所有渲染器直接从 `s.x, s.y` 读取位置并绘制。没有位置lerp/平滑跟随。由于逻辑层每帧移动单位(60fps时每帧约1-3px),移动看起来像**离散跳跃的停顿帧序列**而非连续运动。火柴人步行循环由移动距离驱动,但由于位置离散,动画和位移不同步,造成**滑步感**。

---

#### 视3: 零死亡动画 — 单位瞬间消失

**文件:** `js/combat.js` 第 412行 `killSoldier()`

```js
target.alive = false;  // 瞬间移除
// 只有一条 ring + 一个 "击破" fx 文本,无动画
```

没有倒下动画、没有消散粒子、没有缩放淡出。同样,合成也没有过渡动画(两个球瞬间变成一个)。

---

#### 视4: 受击反馈仅有短暂闪白(≈0.23s),无受击后摇

**文件:** `js/combat.js` 第 478行, 全部渲染器

- `hitFlash = 0.28`,以 `dt*1.2` 衰减 → 实际可见约0.23秒
- 效果: 单位变白一瞬间 → 立刻恢复
- **缺失:** 击退动画、后仰、震退

快速攻击(芒果弩手0.55s攻速)下受击闪几乎不可见。

---

### 🟡 视觉中等问题

#### 视5: 屏幕震动纯随机无方向性

**文件:** `js/render.js` 的 `draw()` 

```js
const sx = (Math.random() - 0.5) * state.shake * 10;  // 完全随机
```

来自右侧的攻击应让屏幕向左震,但当前各方向等概率。多个源通过 `Math.max` 设置而非累加,轻微震动覆盖之前强震。

#### 视6: 状态效果仅在火柴人模式下有视觉

冰冻冰晶、点燃火焰、减速泥潭、眩晕星星、破甲裂纹 — 全在 `stickman_render_v60.js` 的 `drawStatusFX()` 中。如果单位走其他渲染路径,状态效果**完全不可见**。

#### 视7: 全兵种统一步行循环 — 闲置时无差异化

4帧步行循环对所有20种水果兵完全相同。区别仅在于武器和头部emoji。重装坦克步履应与刺客不同。

#### 视8: 攻击特效是简单直线,无刀光弧线

攻击效果是从攻击者到目标的直线。没有弧形斩击、扇形冲击波、武器拖尾。

#### 视9: HP条可见性不一致

| 渲染器 | HP条规则 |
|--------|---------|
| `render.js` | HP<100%才显示 |
| `battle_skin.js` | HP<98.5%或受击闪或护盾>0才显示 |
| `combat_clarity.js` | 始终显示 |
| `stickman_render_v60.js` | 委托给battle_skin |

同一帧中,满血单位可能在一种渲染器显示HP条,另一种不显示。

#### 视10: 绘制顺序不尊重Y轴深度 — 后排遮挡前排

单位按所属方批量绘制(先全部player,后全部enemy),而非按Y位置排序。Y较小的前排和Y较大的后排以任意顺序重叠。

## 📝 第4轮勘误与补充

### 数值修正

**问题1 (后排安全锚):** 初始计算中 `fieldBottom()` 误取为538,实际为526。修正后的关键数值:

| 参数 | 原报告值 | 修正值 |
|------|---------|--------|
| fieldBottom() | 538 | **526** |
| safeAnchor | 496 | **484** |
| 平衡点Y | 435 | **423** |
| 离敌墙距离 | ~55px | **~145px** |
| 中段后退速度 | 27.7 px/s | **21.6 px/s** |

定性结论不变: 后排单位仍永远无法到达敌方城墙。实际上145px的差距比原先估计的55px更大。

**问题4 (Y-clamp):** 不仅影响玩家方攻城单位,敌方攻城单位同样受影响:
- 玩家攻城 Y=278 → clamp → **296** (上弹18px)
- 敌方攻城 Y=546 → clamp → **526** (下弹20px)

### 结论限定

**问题11 (近战范围):** "所有近战范围=24"的表述过于宽泛。实际情况:

| 角色 | 单位 | range属性 | fruitRange返回值 | 受影响? |
|------|------|-----------|-----------------|---------|
| tank | 西瓜盾卫/椰子守卫/草莓骑士/牛油果力士 | melee | **24** | ✅ 有6px死区 |
| front | 菠萝枪兵/火龙果战士 | mid | **42** | ❌ 不受影响 |
| rush | 香蕉突击/柠檬刺客/橄榄刺客 | melee | **24** | ✅ 有6px死区 |

6px死区(24-30)仅影响 tank 和 rush,不影响 front。front 单位可在30px处攻击。结论方向不变但范围收窄。

### 补充: 南瓜滚轮受影响最严重

南瓜滚轮(role=siege, range=melee)同时受**问题1**(后排安全锚,无前排时停在Y=423)和**问题5**(有前排攻城时停在Y=336)影响。它的核心机制是"滚到敌墙爆炸",但在当前逻辑下**永远无法到达敌墙**,其死亡滚动被动只在自身被击杀时触发。这意味着该单位的设计意图被完全阻断。
