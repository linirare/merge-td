## P0.5 — 新增安全问题（3 项）

### 16. 全 API 共享同一个 rate limiter：200次/15分钟
- **证据**: `server/index.js:27-28` — 单个 `apiLimiter` 挂载在所有 `/api/` 路由上。登录、注册、存档保存、聊天、邮件**共用同一个计数器**
- **影响**: 
  - 一个活跃玩家频繁 `saveAll()`（product_shell.js 中有 8 处调用）可快速耗尽 200 次配额 → 封杀**所有用户**的登录、注册
  - 登录爆破限速被绕过：攻击者可先消耗正常 API 呼叫，耗尽配额后 login 也无法使用，但对攻击者无区别
- **修法**: 拆分为多个 limiter：`authLimiter` (登录注册，20次/15min) + `apiLimiter` (游戏操作，500次/15min)

### 17. 无 SIGINT 处理（Windows Ctrl+C 不能优雅关闭）
- **证据**: `grep -c SIGINT server/index.js` → **0**. 仅有 `SIGTERM` 处理（`server/index.js:231`）
- **影响**: Windows 下 Ctrl+C 发送 SIGINT 而非 SIGTERM。按 Ctrl+C 后服务器硬终止，DB 写入可能截断，PvP 连接不清理
- **修法**: `index.js` 加 `process.on('SIGINT', ...)` 复用同一 handler

### 18. tools/ 目录含 2 个源码 patcher 脚本 — 误运行会破坏文件
- **证据**:
  - `tools/restore_juice.js` — 对 `js/juice.js` 做字符串替换插入代码，**重复运行会重复插入**
  - `tools/rewrite_weapons.js` — 重写 `js/stickman_render_v61.js` 的函数，使用硬编码行号
- **修法**: 确认当前已 patch 后标记为不可运行，或改为一键脚本加幂等检查

## P1.2 — PvP 系统问题（11 项）

### 20. pvp-sim.js setTimeout/setInterval 替换为 no-op → 异步代码静默失败
- **证据**: `server/pvp-sim.js:235` — 沙箱 `buildSandbox` 中用 `()=>{}` 替换了 `setTimeout`/`setInterval`
- **影响**: 如果战斗引擎中任何代码将来使用了异步定时（如延迟效果、渐变动画、超时检测），会在沙箱内**静默跳过**，无警告无报错
- **修法**: 覆写为 `() => { console.warn('async call ignored in sandbox') }` 或直接移除（当前引擎纯同步，无任何调用）

### 21. pvp.js 残留 p2p 死代码路径
- **证据**: `js/pvp.js:418-449` — `applyRemoteAction` 函数中有一段代码被注释为 "服务器权威下已不发，保留兜底无害"。该路径在服务器权威模式下永远不会被执行
- **影响**: 误导后续维护者，增加代码理解成本。无实际功能影响
- **修法**: 删除这段死代码，保留函数签名作为未来扩展点即可

### 22. showPvpDisconnectResult 使用 cloneNode 破坏事件监听
- **证据**: `js/pvp.js:507-524` — `showPvpDisconnectResult` 使用 `cloneNode(true)` 复制 DOM 元素来显示结果。`cloneNode` 只复制 DOM 属性，**不复制通过 addEventListener 绑定的事件处理函数**
- **影响**: 断开连接弹窗可能无法正常交互（按钮点击无响应），具体取决于事件绑定方式
- **修法**: 创建新 DOM 元素代替 clone，或者保留原始节点引用

### 23. 🔴 CRITICAL: 客户端可指定任意召唤消耗（免费出兵漏洞）
- **证据**: `server/pvp-sim.js:142` → `spendJuice(sideName, Number(p.cost || actionCost(sideName)));` — `p.cost` 直接来自客户端的 action payload（经 `normalizeActionPayload` 校验后保留为 `cost` 字段）。客户端发送 `cost: 1`，服务端就只扣 1 SP
- **影响**: 攻击者修改客户端代码，所有 summon/urgent_dispatch 动作都发送 `cost: 1`，即可始终以 1 SP 的代价召唤士兵（正常召唤成本随次数递增至 12 SP）。**相当于免费出兵**，彻底破坏 PvP 公平性
- **修法**: `applyAction` 中永远使用 `actionCost(sideName)` 计算实际消耗，完全忽略 `p.cost`

### 24. PvP Observer 无 per-IP 限制
- **证据**: `server/pvp-server.js:419` — observer 上限 20 人，但无 per-IP/per-UID 限制
- **影响**: 单个攻击者可创建 20 个 observer 连接耗尽服务器资源（每个 observer 有独立的 WebSocket 连接和消息队列）
- **修法**: 加 per-IP 限制（2-3 个 observer/IP）+ per-room observer 上限保持 20

### 25. PvP 聊天消息双重持久化
- **证据**: `server/pvp-server.js:427-445` — 聊天消息同时 push 到内存数组 `chatHistory` 和 `chat_logs` 数据库表
- **影响**: 冗余写入。`chatHistory` 仅用作最近 200 条的内存缓存，`chat_logs` 是完整持久化。每次聊天都写一次 DB 增加了 I/O
- **修法**: 移除 DB 写入（PVP 聊天不需要永久保存），或改为批量写入（每 30s flush 一次）

### 26. 🔴 CRITICAL: WebSocket 无连接频率/数量限制
- **证据**: `server/pvp-server.js:476-482` — `attachSocket` 将每个连接加入 `allClients`，无任何 per-IP 或全局连接上限。`makeRoomId()` (line 20-25) 仅 9000 个可用房间码，无最大房间数限制
- **影响**: 
  - 单个 IP 可开无限 WebSocket 连接 → 服务器内存/句柄耗尽
  - 每个连接可创建独立房间 → 无限房间占内存
  - 房间码碰撞时 `do...while` 无重试上限 → 极端情况死循环
- **修法**: 加 `connectionIpLimiter`（同 IP 最多 5 连接）+ 全局最大房间数（200）+ `makeRoomId` 最大重试次数（100 次后报错）

### 27. MEDIUM: 客户端城墙血量初始化与服务端不一致（首帧视觉跳跃）
- **证据**: 服务端沙箱 `pvp-sim.js:54` → `WALL = 72`（取自 `js/config.js:276` 的 `BASE_WALL_HP = 72`）。客户端 `js/pvp.js:235-237` → `BASE_WALL_HP + getWallBonus(meta)`（含养成加成，如 72 + 48 = 120）
- **影响**: 对局开始后约 66ms（2 帧）首帧快照到达，客户端城墙血量从 120 突降到 72。虽然服务端是权威不影响胜负，但视觉上"城墙缩水"让玩家困惑
- **修法**: `pvp.js startPvpMatch` 中在等第一帧快照前，暂用 `BASE_WALL_HP` 显示，或服务端 PvP 也应用 `getWallBonus`

### 28. MEDIUM: PvP 无断线重连机制
- **证据**: `js/pvp.js:52-53` — `close` 事件仅设置状态"连接已断开"，无自动重连或会话恢复逻辑
- **影响**: 短暂网络抖动（<1s）即导致对局终结。重连后无法恢复进行中的比赛，房间和战场状态丢失
- **修法**: 添加 WebSocket 自动重连（指数退避），服务端 room 保留状态 30s 等待重连

### 29. LOW: 观察者(observer)可实时看到双方棋盘（观战作弊）
- **证据**: `server/pvp-server.js:124-127` — `broadcastRoom` 将包含双方棋盘(p+e)的完整快照发给 observers。没有延迟或只给一方视角
- **影响**: 观察者可通过场外语音实时通报对方排兵布阵，破坏公平性
- **修法**: 观察者只接收延迟 5s 的快照，或只接收某一方的视角

### 30. LOW: WebSocket 分片帧静默丢弃
- **证据**: `server/pvp-server.js:57-96` — `decodeFrames` 仅处理 opcode 0x1(text)、0x8(close)、0x9(ping)。Continuation frame (0x0) 和 binary (0x2) 静默忽略
- **影响**: 如果客户端（或中间代理）将一条消息分片发送，只有第一片被处理，后续分片丢失。标准 WebSocket 库不会分片，但恶意或异常客户端可能导致协议错乱
- **修法**: 添加 opcode 0x0 处理（拼接 payload），或记录 warning
